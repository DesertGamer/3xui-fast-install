Эта инструкция предназначена для ИИ-агента, который помогает пользователю установить проект на VPS.

**Первый вопрос, который агент обязан задать до любых других действий:**

Только после ответа пользователя двигайся дальше по инструкции. Не спрашивай IP или домен, пока не убедился, что они у пользователя есть.

### Если домена нет

Предложи бесплатный **DuckDNS**:

1. Перейти на [duckdns.org](https://www.duckdns.org/) и авторизоваться (через Google или GitHub)
2. Создать домен, например `myvpn.duckdns.org`
3. Указать IP сервера
4. A-запись активируется за несколько секунд

После этого попроси у пользователя или найди в контексте:

- IP сервера: например `1.2.3.4`
- Домен: например `myvpn.duckdns.org`
- SSH-пользователь: обычно `root`
- SSH-порт: обычно `22`
- SSH-ключ, если нужен: например `-i ~/.ssh/id_rsa`

Требования к серверу:

- Debian 12+ или Ubuntu 22.04+
- root-доступ по SSH
- A-запись домена указывает на IP сервера
- открыты порты `22/tcp`, `80/tcp`, `443/tcp`, `63000/udp` или их кастомные значения из env

## Правила безопасности

> **СТОП. Прочитай это перед любым действием на сервере.**

### Единственный разрешённый способ что-либо сделать — запустить скрипт

Всё — нативные сервисы `x-ui` и `caddy`, конфиги, БД, сертификаты, маршрутизация — создаётся **только через скрипты**:

- установка → `deploy.sh`
- отдельный шаг → `steps/<name>.sh` через `ssh root@IP 'bash /root/3xui-setup/<name>.sh'`
- бэкап → `backup.sh`
- восстановление → `restore.sh`
- удаление → `uninstall.sh`

### Что запрещено делать — всегда, без исключений

Следующие действия **запрещены**, даже если кажется, что это самый быстрый путь:

- ручная правка `/etc/caddy/Caddyfile` или `/var/www/html` в обход скрипта
- `nano`, `vim`, `sed`, `echo >`, `tee` для редактирования файлов на сервере
- `systemctl start/stop/restart x-ui` или `caddy` вручную для правки конфигурации
- `sqlite3` напрямую для изменения базы `/etc/x-ui/x-ui.db`
- создание файлов сертификатов, compose-файлов, конфигов руками
- любая команда `rm -rf`, `git reset --hard` без явного запроса пользователя

### Что делать вместо этого

Если что-то не работает:

1. **Читай логи — не действуй:**
   ```bash
   ssh root@<IP> 'tail -n 200 /root/3xui-install-full.log'
   ```
2. **Покажи лог пользователю и объясни, что именно упало.**
3. **Перезапусти только упавший шаг через скрипт** или весь `deploy.sh`.
4. Если непонятно что делать — скажи пользователю прямо, не импровизируй.

### Частые нарушения, которых надо избегать

❌ «Caddy работает, давайте запустим следующие шаги вручную»  
✅ Перезапусти `deploy.sh` — он сам разберётся, что уже сделано.

❌ «Файл credentials не создан, давайте запустим шаг с 3x-ui вручную»  
✅ Прочитай лог, найди ошибку, перезапусти через скрипт: `ssh root@<IP> 'DOMAIN=<DOMAIN> bash /root/3xui-setup/xui.sh'`

❌ «Установка не завершена, давайте исправим конфиг x-ui / Caddyfile»  
✅ Никогда не правь конфиг вручную. Перезапусти `deploy.sh`.

## Проверки перед запуском

Проверь локально:

```bash
bash -n deploy.sh backup.sh restore.sh scripts/local_lib.sh
bash -n steps/*.sh
```

Проверь SSH:

```bash
ssh root@<IP> 'echo ok'
```

Проверь DNS:

```bash
dig +short A <DOMAIN>
```

Если `dig` недоступен локально, можно проверить на сервере после подключения:

```bash
getent ahosts <DOMAIN>
```

## Основной запуск

### Шаг 1 — Собери все параметры перед запуском

Перед тем как запускать `deploy.sh`, убедись, что у тебя есть все четыре значения:

| Параметр   | Пример                      | Как передаётся                      |
| ---------- | --------------------------- | ----------------------------------- |
| IP сервера | `1.2.3.4`                   | первый аргумент скрипта             |
| Домен      | `myvpn.duckdns.org`         | переменная `DOMAIN=` перед командой |
| SSH-порт   | `22` (по умолчанию)         | переменная `SSH_PORT=`, если не 22  |
| SSH-ключ   | `~/.ssh/id_rsa` (если есть) | флаг `-i` после IP                  |

> ⚠️ Без `DOMAIN=` скрипт спросит домен интерактивно. Всегда передавай его явно через переменную — это надёжнее.

### Шаг 2 — Собери команду

Стандартный случай (порт 22, по паролю):

```bash
DOMAIN=myvpn.duckdns.org bash deploy.sh 1.2.3.4
```

С SSH-ключом:

```bash
DOMAIN=myvpn.duckdns.org bash deploy.sh 1.2.3.4 -i ~/.ssh/id_rsa
```

Нестандартный SSH-порт:

```bash
SSH_PORT=2222 DOMAIN=myvpn.duckdns.org bash deploy.sh 1.2.3.4
```

Нестандартный порт + SSH-ключ:

```bash
SSH_PORT=2222 DOMAIN=myvpn.duckdns.org bash deploy.sh 1.2.3.4 -i ~/.ssh/id_rsa
```

### Шаг 3 — Покажи команду пользователю и дождись подтверждения

Перед запуском выведи собранную команду и попроси пользователя подтвердить. Не запускай молча.

### Шаг 4 — Запусти и следи за выводом

`deploy.sh` будет выводить прогресс в реальном времени. Подробный лог пишется в `/root/3xui-install-full.log`.

Если скрипт задаёт интерактивный вопрос (бэкап, SSH-ключ) — не отвечай автоматически. Спроси пользователя.

### Шаг 5 — Проверь результат

После завершения `deploy.sh` выведет содержимое `/root/3xui-credentials.txt` с URL панели и ссылкой подписки. Покажи их пользователю.

## Что считается успехом

На сервере должны быть:

```bash
systemctl is-active x-ui     # активный сервис панели 3x-ui (нативно)
systemctl is-active caddy    # активный сервис Caddy selfsteal (нативно)
```

Ожидаемые сервисы:

- `x-ui` — systemd-сервис панели 3x-ui (нативная установка)
- `caddy` — systemd-сервис Caddy для сертификата и маскировки (нативная установка)

Файлы:

- `/root/3xui-credentials.txt`
- `/root/3xui-install.log`
- `/root/3xui-install-full.log`
- `/etc/x-ui/x-ui.db`
- `/usr/local/x-ui/` (бинари x-ui и xray)
- сертификат — в каталоге Caddy `/var/lib/caddy/.local/share/caddy/certificates/…/<DOMAIN>/` (его выпускает и продлевает Caddy; xray читает напрямую оттуда, копии в `/root/cert` больше нет)

Панель (на `443` через Caddy, без отдельного порта):

```text
https://<DOMAIN>/<PANEL_PATH>/
```

> Панель, подписки и сайт-маскировка отдаются на одном порту `443`: xray (Reality) на 443 отдаёт не-Reality трафик в Caddy (`:9443`), а Caddy проксирует `/<PANEL_PATH>/` и `/<SUB_PATH>` на локальный x-ui (`127.0.0.1:60000/60001`), остальное — на сайт-заглушку. Если xray лёг, панель временно недоступна на 443, но жива на `127.0.0.1:60000` — чини через `systemctl restart x-ui` или SSH-туннель `ssh -L 8443:127.0.0.1:60000 root@<IP>`.

Клиент:

- первый клиент создан автоматически сразу в VLESS Reality и Hysteria2
- ссылка подписки должна быть в `/root/3xui-credentials.txt`

## Диагностика ошибок

Если `deploy.sh` упал, прочитай полный лог:

```bash
ssh root@<IP> 'tail -n 200 /root/3xui-install-full.log'
```

И краткий лог:

```bash
ssh root@<IP> 'cat /root/3xui-install.log'
```

Проверь сервисы панели и Caddy:

```bash
ssh root@<IP> 'systemctl --no-pager status x-ui caddy'
```

Проверь все сервисы:

```bash
ssh root@<IP> 'systemctl --no-pager status x-ui caddy warp-svc opera-proxy tor fail2ban'
```

Проверь firewall:

```bash
ssh root@<IP> 'ufw status verbose'
```

## Частые проблемы

### SSH host key changed

Симптом:

```text
REMOTE HOST IDENTIFICATION HAS CHANGED
Host key verification failed.
```

`deploy.sh` автоматически обнаруживает это и спрашивает:

```text
Удалить старый ключ и продолжить? [Y/n]
```

Как агент: уточни у пользователя, переустанавливался ли сервер. Если да — отправь Enter или `y`, скрипт сам выполнит `ssh-keygen -R <IP>` и повторит подключение. Если сервер не переустанавливался — отправь любую другую клавишу, скрипт прервётся, и нужно разбираться отдельно.

```bash
ssh-keygen -R <IP>
DOMAIN=<DOMAIN> bash deploy.sh <IP>
```

### DNS не указывает на сервер

Симптом в логе:

```text
DNS для <DOMAIN> указывает на <OLD_IP>, а не на этот сервер
```

Действия:

1. Попроси пользователя исправить A-запись домена.
2. Дождись обновления DNS.
3. Повтори:

```bash
DOMAIN=<DOMAIN> bash deploy.sh <IP>
```

### Selfsteal падает без подробной ошибки

Симптом:

```text
[SELFSTEAL] ❌ Script terminated with error code: 1
```

Действия:

1. Убедись, что в `steps/selfsteal.sh` внешний installer запускается с `--debug`.
2. Скопируй актуальный шаг на сервер:

```bash
scp steps/_lib.sh steps/selfsteal.sh root@<IP>:/root/3xui-setup/
```

3. Повтори только selfsteal:

```bash
ssh root@<IP> 'DOMAIN=<DOMAIN> bash /root/3xui-setup/selfsteal.sh'
```

4. Если selfsteal прошёл, продолжи установку с шага `xui.sh` или перезапусти `deploy.sh`.

### Сертификат не выдан

Симптом:

```text
Caddy не получил сертификат за 60 секунд
```

Если установка зависает на этапе Selfsteal, сначала посмотри подробные логи Caddy (journald):

```bash
ssh root@<IP> 'journalctl -u caddy --no-pager -n 150'
```

Если сервис только что стартовал и нужен живой поток логов:

```bash
ssh root@<IP> 'journalctl -u caddy -f'
```

Дополнительно проверь:

```bash
ssh root@<IP> 'systemctl --no-pager status caddy'
ssh root@<IP> 'ufw status verbose'
dig +short A <DOMAIN>
```

Частые причины:

- порт `80/tcp` закрыт у провайдера или firewall
- DNS ещё не обновился
- домен указывает не на этот IP
- превышен rate limit
- неправильный домен в переменной `DOMAIN`

### 3x-ui не стартует

Проверь:

```bash
ssh root@<IP> 'systemctl --no-pager status x-ui'
ssh root@<IP> 'journalctl -u x-ui -n 120 --no-pager'
```

Если нет БД:

```bash
ssh root@<IP> 'ls -la /etc/x-ui /etc/x-ui/x-ui.db'
```

## Авто-бэкап при повторном деплое

Если на сервере уже есть `/root/3xui-credentials.txt`, `deploy.sh` обнаружит это и спросит:

```text
Создать бэкап перед деплоем? [Y/n]
```

Enter или `y` — запустится `backup.sh`, архив сохранится в `backups/`. Только `n` пропустит бэкап. Если бэкап не удался, скрипт спросит отдельно, продолжать ли без него.

Как агент: не отвечай автоматически на этот запрос без подтверждения пользователя, если бэкап не был явно запрошен.

## Backup и restore

Используй `backup.sh` и `restore.sh` как локальные команды из корня репозитория. Они подключаются к серверу по SSH, поэтому поддерживают те же параметры доступа: `SSH_USER`, `SSH_PORT` и дополнительные SSH-аргументы после IP, например `-i ~/.ssh/id_rsa`.

### Когда делать backup

Предлагай или выполняй backup перед:

- повторным запуском установки на уже настроенном сервере
- ручным исправлением `/etc/x-ui/`, `/usr/local/x-ui/`, `/etc/caddy/`, `/var/www/html/` или `/var/lib/caddy/`
- восстановлением из старого архива
- обновлением скриптов, если есть риск затронуть текущую конфигурацию

Команды:

```bash
bash backup.sh <IP>
bash backup.sh <IP> -i ~/.ssh/id_rsa
SSH_PORT=2222 bash backup.sh <IP>
BACKUP_DIR=~/backups bash backup.sh <IP>
```

Что делает `backup.sh`:

- собирает архив на сервере (копирует `/etc/x-ui`, сертификаты, Caddy-конфиг и др.)
- скачивает архив локально в `backups/` или в `BACKUP_DIR`
- удаляет временный архив из `/tmp` на сервере

Архив должен содержать:

- `etc-x-ui/` — база и настройки 3x-ui (`/etc/x-ui`)
- `cert/` — сертификаты
- `caddy/` — selfsteal Caddy без логов
- `3xui-credentials.txt` — файл доступов, если он есть

После backup сообщи пользователю путь к архиву и размер файла. Если команда упала, проверь SSH-доступ и свободное место:

```bash
ssh root@<IP> 'df -h / /tmp; systemctl is-active x-ui caddy; ls -la /etc/x-ui /etc/caddy /var/www/html 2>/dev/null'
```

### Когда делать restore

Используй `restore.sh`, когда пользователь явно хочет перенести или откатить состояние 3x-ui из существующего локального архива.

Перед restore обязательно:

1. Проверь, что архив существует локально:

```bash
ls -lh <backup.tar.gz>
```

2. Уточни у пользователя, что текущие данные на сервере можно заменить.
3. Если на сервере есть ценная текущая конфигурация, сначала сделай свежий backup.

Команды:

```bash
bash restore.sh <IP> backups/backup_<IP>_<YYYYMMDD_HHMMSS>.tar.gz
bash restore.sh <IP> backups/backup_<IP>_<YYYYMMDD_HHMMSS>.tar.gz -i ~/.ssh/id_rsa
SSH_PORT=2222 bash restore.sh <IP> backups/backup_<IP>_<YYYYMMDD_HHMMSS>.tar.gz
```

`restore.sh` сам запросит подтверждение:

```text
Продолжить? [y/N]
```

Если работаешь как агент и команда ждёт этот ввод, не отвечай `y` автоматически без подтверждения пользователя.

Что делает `restore.sh`:

- загружает архив в `/tmp` на сервер
- останавливает сервисы `x-ui` и `caddy`
- восстанавливает `/etc/x-ui`, `/etc/caddy/Caddyfile`, `/var/www/html`, `/var/lib/caddy` (сертификат) и `/root/3xui-credentials.txt` (старые архивы с `db/`/`cert/` тоже понимает)
- выставляет безопасные права на приватные ключи и файл доступов
- запускает сервисы `caddy` и `x-ui` обратно
- удаляет временный архив с сервера

### Проверки после restore

Проверь сервисы:

```bash
ssh root@<IP> 'systemctl --no-pager status x-ui caddy'
```

Проверь, что файл доступов и БД на месте:

```bash
ssh root@<IP> 'ls -lh /root/3xui-credentials.txt /etc/x-ui/x-ui.db'
```

Проверь логи при ошибках:

```bash
ssh root@<IP> 'journalctl -u x-ui -n 120 --no-pager'
ssh root@<IP> 'journalctl -u caddy -n 120 --no-pager'
```

Если restore переносится на сервер с другим доменом, IP или портами, не считай это обычным восстановлением. Сначала объясни пользователю, что в архиве лежат старые сертификаты, Caddy-конфигурация и URL панели; может потребоваться новая установка или ручная правка конфигов.

## Продолжение после частичного успеха

Если предыдущие шаги уже прошли и упал только финальный шаг `3x-ui`, можно синхронизировать актуальные файлы и запустить только `xui.sh`:

```bash
scp steps/_lib.sh steps/xui.sh root@<IP>:/root/3xui-setup/
ssh root@<IP> 'DOMAIN=<DOMAIN> bash /root/3xui-setup/xui.sh'
```

После ручного запуска `xui.sh` проверь, что существует файл доступов:

```bash
ssh root@<IP> 'cat /root/3xui-credentials.txt'
```

Если файла нет, лучше перезапустить полный `deploy.sh`, потому что именно `setup.sh` создаёт `/root/3xui-credentials.txt` после всех шагов.
